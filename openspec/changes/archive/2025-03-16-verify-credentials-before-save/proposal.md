## Why

首次运行时，系统在验证凭证之前就将用户输入的 app_id 和 token 保存到配置文件。如果用户输入了错误的凭证，这些无效值会被持久化，导致 API 调用失败，用户必须手动删除配置文件或编辑配置文件才能修正。

这是一个糟糕的用户体验——用户在首次使用时就可能遇到障碍。

## What Changes

**核心变更：** 将配置文件的保存延迟到 API 验证成功之后。

- **首次运行流程变更**：收集凭证后暂存于内存，执行命令验证凭证，只有 API 调用成功才保存配置
- **失败处理**：API 验证失败时显示清晰错误信息，不保存配置，用户修正参数后重试命令
- **setup 函数重构**：`runSetup()` 改为返回凭证对象而非直接保存，新增 `collectCredentials()` 函数
- **config 模块扩展**：`loadConfig()` 支持接受内存凭证参数
- **main 函数逻辑变更**：首次运行时先收集凭证到内存，执行命令成功后保存

**用户体验改进：**
- 永不保存无效凭证
- 错误信息清晰，直接指向凭证问题
- 用户只需修正参数重试，无需额外的重置步骤

## Capabilities

### New Capabilities

- `deferred-config-save`: 首次运行时延迟保存配置，直到 API 验证成功后才写入磁盘

### Modified Capabilities

- `config`: "首次运行交互式引导" requirement 需要更新——引导完成后不立即保存配置，而是返回凭证供后续验证

## Impact

**代码变更：**
- `src/setup.ts` - 拆分 `runSetup()` 为 `collectCredentials()` + 保存逻辑
- `src/index.ts` - main 函数首次运行流程重构
- `src/config.ts` - `loadConfig()` 支持内存凭证参数

**行为变更：**
- 首次运行时，如果 API 凭证无效，配置文件不会被创建
- 错误提示中会明确说明配置未保存，请用户重试

**不兼容变更：**
- `runSetup()` 函数签名变更（如果外部代码依赖）——本项目内部使用，无外部依赖
