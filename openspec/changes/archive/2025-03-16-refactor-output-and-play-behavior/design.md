## Context

当前 tts-cli 使用 `--play` 和 `--output` 参数控制运行行为，但存在以下问题：

1. **语义混淆**：`--play` 参数既控制是否播放，又隐式控制是否保存（当无 `--output` 时不保存）
2. **重复生成**：每次运行都调用 API，即使目标 MP3 文件已存在
3. **路径限制**：`--output` 必须指定完整文件路径，不支持文件夹路径
4. **配置冗余**：`output.dir` 配置项实际上未被使用，增加复杂度

用户期望更清晰的行为：
- 无参数时：保存到默认位置
- `--play`：播放（如果已存在则播放本地，否则生成后播放）
- `--output`：支持文件夹或文件路径
- 文件已存在时：有明确的覆盖机制

## Goals / Non-Goals

**Goals:**
- 重新定义 `--play` 参数语义：仅控制是否播放，不再控制是否保存
- 默认行为改为"总是保存 MP3"
- 添加 `--force` 参数支持强制覆盖
- `--output` 支持文件夹路径（使用 `fs.stat()` 检测）
- 文件已存在时播放本地 MP3（避免重复调用 API）
- 删除未使用的 `output.dir` 配置项

**Non-Goals:**
- 修改音频编码参数（采样率、比特率等）
- 修改 API 调用逻辑
- 改变错误处理机制

## Decisions

### 决策 1: 总是保存 MP3

**选择**：移除"只播放不保存"模式，所有运行模式都会保存 MP3。

**理由**：
- 用户通常希望保留生成的音频文件
- 简化参数语义（`--play` 仅控制播放）
- 避免"我刚才播放的音频去哪了"的困惑

**权衡**：
- 可能产生用户不需要的文件
- 缓解：文件存在时提供明确的提示和覆盖机制

### 决策 2: 使用 `fs.stat()` 检测路径类型

**选择**：`--output` 参数使用 `fs.stat()` 来区分文件路径和文件夹路径。

**理由**：
- 自动检测比要求用户添加结尾 `/` 更用户友好
- 符合 Unix 工具惯例（如 `cp file dir/`）

**实现逻辑**：
```typescript
if (outputPath) {
  const stat = await fs.stat(outputPath).catch(() => null);
  if (stat?.isDirectory()) {
    // 文件夹路径：使用输入文件名
    const inputBasename = path.basename(inputPath, path.extname(inputPath));
    finalPath = path.join(outputPath, `${inputBasename}.mp3`);
  } else {
    // 文件路径（可能不存在）
    finalPath = outputPath;
  }
}
```

### 决策 3: 文件存在时的行为分层

**选择**：根据参数组合决定文件存在时的行为：

| 参数组合 | 文件存在 | 行为 |
|---------|---------|------|
| 无 | ✓ | 提示已存在，退出 |
| `--play` | ✓ | 播放本地 MP3 |
| `--force` | ✓ | 强制重新生成 |
| `--play --force` | ✓ | 强制重新生成并播放 |

**理由**：
- 默认安全：不覆盖已存在文件
- `--play` 作为"快捷播放"：直接播放本地文件
- `--force` 作为显式覆盖：用户明确知道会覆盖

### 决策 4: 本地 MP3 播放使用 ffplay

**选择**：播放本地 MP3 时直接调用 `ffplay <file>`，不使用 PCM 流。

**理由**：
- 简单直接，无需转码
- ffplay 原生支持 MP3
- 与生成模式的 PCM 流播放区分开

**命令**：
```bash
ffplay -nodisp -autoexit <local_mp3_file>
# -nodisp: 不显示视频窗口
# -autoexit: 播放完自动退出
```

### 决策 5: 静默覆盖（使用 --force 时）

**选择**：`--force` 参数静默覆盖已存在文件，不询问确认。

**理由**：
- CLI 工具应该支持脚本化，不应有交互式确认
- `--force` 本身就是显式的覆盖意图
- 保持与 Unix 工具（如 `rm -f`）的一致性

## Risks / Trade-offs

### 风险 1: 内存占用（PCM 边播边存）

**问题**：当前实现在 `--play` 模式下将 PCM 数据累积在内存中，长音频可能占用大量内存。

**估算**：
- PCM 格式：16bit + 单声道 + 24000Hz = 48 KB/秒
- 1小时音频 ≈ 172 MB RAM

**缓解**：
- 现代电脑内存充足，170MB 可接受
- 如果未来需要优化，可考虑临时写入 `.pcm` 文件

### 风险 2: 用户误覆盖文件

**问题**：用户可能误用 `--force` 覆盖重要文件。

**缓解**：
- `--force` 需要显式指定
- 文件存在时默认不覆盖，有明确提示
- 提示信息中包含播放本地的建议

### 风险 3: 配置迁移

**问题**：删除 `output.dir` 后，旧配置文件可能包含该字段。

**缓解**：
- `loadConfig()` 可安全忽略未知字段
- 无需强制迁移，自然淘汰

## Migration Plan

无需数据迁移，纯行为变更：

1. **部署**：
   - 更新代码后直接部署
   - 旧配置文件继续工作（忽略 `output.dir`）

2. **用户适应**：
   - 更新文档说明新的参数行为
   - 在 README 中添加行为对照表

3. **回滚**：
   - 如需回滚，恢复旧代码即可
   - 配置文件变更向后兼容
